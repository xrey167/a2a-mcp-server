A2A-MCP Server

Ein Multi-Protokoll-Automatisierungs-Framework, das Claude Code, Zed IDE und A2A-kompatible Agenten mit einer Flotte von 14 spezialisierten Worker-Agents verbindet.

MCP (Anthropic) A2A (Google) ACP (Zed IDE) 14 Worker-Agents 50+ MCP-Tools Bun · TypeScript · MIT

Überblick

Der A2A-MCP Server ist ein Single-Entry-Point-Orchestrator (src/server.ts), der gleichzeitig als MCP-Server (stdio), A2A-HTTP-Endpunkt (Port 8080) und ACP-Server (stdin/stdout) fungiert. Beim Start spawnt er lokale Worker-Prozesse, entdeckt ihre Fähigkeiten per GET /.well-known/agent.json und baut eine dynamische Routing-Map auf.

Das System ist für drei Nutzergruppen konzipiert:

  • KI-Assistenten (Claude Code, Zed): nutzen MCP/ACP-Protokoll, um Aufgaben an Worker zu delegieren
  • Externe Dienste: sprechen A2A-HTTP oder triggern Webhooks
  • Entwickler: erweitern das System mit eigenen Workers, Pipelines und Plugins
Schlüsselprinzip: stdout ist reserviert Alle Log-Ausgaben müssen über process.stderr.write() erfolgen — niemals console.log(). stdout ist ausschließlich für den MCP JSON-RPC-Stream.

Quick Start

Voraussetzungen

  • Bun ≥ 1.0
  • ANTHROPIC_API_KEY — für ask_claude und LLM-basiertes Routing
  • Optional: GOOGLE_API_KEY (Design-Worker), NASA_FIRMS_KEY (Climate-Worker)

Installation & Registrierung

# Repository klonen und Abhängigkeiten installieren
git clone https://github.com/xrey167/a2a-mcp-server
cd a2a-mcp-server
bun install

# Konfiguration initialisieren (Profil wählen)
bun src/cli.ts init           # full  — alle 14 Worker
bun src/cli.ts init --lite    # lite  — shell + web + ai
bun src/cli.ts init --data    # data  — lite + data
bun src/cli.ts init --osint   # osint — lite + 6 OSINT-Worker

# Mit Claude Code registrieren (user scope)
claude mcp add --scope user a2a-mcp-bridge -- bun $(pwd)/src/server.ts
claude mcp list   # Verbindung prüfen

Starten

# Orchestrator + alle aktivierten Worker starten
bun src/server.ts

# Einen einzelnen Worker isoliert starten
bun src/workers/shell.ts

# Tool testen ohne Claude Code neu zu starten
env -u CLAUDECODE claude -p "Use the delegate tool to run echo hello" \
  --allowedTools "mcp__a2a-mcp-bridge__delegate"

Profile

Das System unterstützt Profile, um gezielt nur eine Teilmenge der Worker zu starten. Das aktive Profil wird in ~/.a2a-mcp/config.json gesetzt.

full
Alle 14 Worker aktiv. Maximale Fähigkeiten.
lite
shell + web + ai. Minimaler Ressourcenverbrauch.
data
lite + data. Für Datenanalyse-Workflows.
osint
lite + 6 OSINT-Worker (news, market, signal, monitor, infra, climate).
// ~/.a2a-mcp/config.json
{
  "profile": "lite",
  "workers": { "enabled": ["shell", "web", "ai"] },
  "remoteWorkers": [
    { "url": "https://my-agent.example.com", "name": "custom-agent" }
  ]
}

Systemarchitektur

Der Orchestrator ist das Herzstück. Er nimmt Anfragen über drei Protokolle entgegen, wendet Sicherheits- und Observability-Schichten an und leitet sie an den passenden Worker weiter.

Startup-Sequenz

  1. 1
    Konfiguration laden

    Liest ~/.a2a-mcp/config.json: Profil, aktivierte Worker, Remote-Worker-URLs.

  2. 2
    Worker spawnen

    Startet aktivierte lokale Worker als Subprozesse via Bun.spawn.

  3. 3
    Agent Discovery

    Fragt jeden Worker per GET /.well-known/agent.json ab (Exponential Backoff, bis zu 5 Versuche).

  4. 4
    Routing-Map aufbauen

    Erstellt die skillId → workerURL-Map für schnelles Routing.

  5. 5
    Health Polling

    Alle 30 Sekunden werden Worker-Health-Checks durchgeführt. Circuit Breaker reagieren bei Ausfällen.

Skill-Routing

Das delegate-Tool ist das primäre MCP-Werkzeug. Es leitet Anfragen nach einer dreistufigen Priorisierung weiter:

  1. 1
    Direktes Routing via agentUrl

    Ist eine explizite URL angegeben, wird die Anfrage direkt (circuit-broken) dorthin gesendet.

  2. 2
    Skill-Map-Lookup via skillId

    Ist eine Skill-ID bekannt, findet das System den zuständigen Worker über die interne Map.

  3. 3
    LLM-basiertes Routing

    Kein explizites Ziel? ask_claude analysiert die Worker-Cards und wählt {url, skillId}.

Capability Negotiation src/capability-negotiation.ts ermöglicht versions-bewusstes Routing mit SemVer-Matching, Load-Awareness und Health-Scores.

Eigenen Worker erstellen

# Scaffold generieren
bun src/cli.ts create-worker my-tool

Die neue Datei src/workers/my-tool.ts muss folgendes implementieren:

  1. Einen Fastify-Server auf einem freien Port
  2. Ein AGENT_CARD-Export mit Skill-Definitionen
  3. Einen Eintrag im ALL_WORKERS-Array in src/server.ts
  4. Alle Ausgaben ausschließlich via stderr

Workflow Engine

src/workflow-engine.ts führt multi-step DAG-basierte Workflows aus. Steps deklarieren Abhängigkeiten; die Engine führt sie in topologischer Reihenfolge mit maximaler Parallelität aus.

Kernkonzepte

  • Template-Referenzen: {"{{stepId.result}}"} verbindet Outputs mit Inputs
  • Bedingte Ausführung: when-Ausdrücke pro Step
  • Fehlerstrategien: fail, skip, retry pro Step
// Beispiel-Workflow: Datei lesen → analysieren → Bericht erstellen
const workflow: WorkflowDefinition = {
  steps: [
    {
      id: "read",
      skillId: "read_file",
      args: { path: "./data.csv" }
    },
    {
      id: "analyze",
      skillId: "analyze_data",
      dependsOn: ["read"],
      args: { data: "{{read.result}}" }
    },
    {
      id: "report",
      skillId: "ask_claude",
      dependsOn: ["analyze"],
      args: { prompt: "Erstelle einen Bericht aus: {{analyze.result}}" }
    }
  ]
};

Sandbox Execution

sandbox_execute führt TypeScript-Code in isolierten Bun-Subprozessen aus. Der Code hat Zugriff auf alle Worker-Skills via skill(id, args).

  • Variablen persistieren per Session in SQLite (~/.a2a-sandbox.db)
  • Ergebnisse >4KB werden automatisch in FTS5 indiziert → durchsuchbar via search(varName, query)
  • sandbox_vars verwaltet persistierte Variablen
// Im Sandbox-Kontext verfügbar:
const result = await skill("ask_claude", { prompt: "Analysiere ..." });
const found = await search("result", "Schlüsselwort");
$vars.myData = result; // persistieren

Agent Collaboration

src/agent-collaboration.ts implementiert Multi-Agent-Konsens- und Verhandlungsprotokolle.

StrategieBeschreibung
fan_outParallele Abfrage mehrerer Agents + Merge der Ergebnisse
consensusKI-bewertetes Voting über Agent-Antworten
debateIterative Kritik/Verfeinerung zwischen Agents
map_reduceAufgabe verteilen → Teilergebnisse aggregieren

Das MCP-Tool collaborate gibt eine taskId zurück — das Ergebnis wird asynchron abgerufen.

Event Bus

src/event-bus.ts implementiert ein Real-Time Pub/Sub-System mit topic-basiertem Routing.

  • Wildcards: * (ein Segment), # (multi-Segment) — z.B. agent.*.completed, workflow.#
  • Event History mit konfigurierbarer Retention
  • Dead Letter Queue für fehlgeschlagene Deliveries
  • Events werden automatisch bei Agent-Completion/-Failure publiziert
MCP-ToolFunktion
event_publishEvent publizieren
event_subscribeTopic abonnieren (mit Wildcard-Support)
event_replayEvents ab einem Timestamp wiederholen

Skill Pipelines (Composer)

src/skill-composer.ts ermöglicht deklaratives Skill-Chaining mit pipe()-Semantik.

  • Output eines Steps wird automatisch Input des nächsten
  • Template-Referenzen: {"{{prev.result}}"}, {"{{input.*}}"}, {"{{steps.alias.result}}"}
  • Fehlerstrategien pro Step: abort (Standard), skip, Fallback-Wert
  • Bedingte Ausführung via when-Ausdrücke
ToolBeschreibung
compose_pipelineNeue Pipeline definieren und speichern
execute_pipelineGespeicherte Pipeline ausführen
list_pipelinesAlle verfügbaren Pipelines auflisten

Project Factory

factory_workflow generiert vollständige Projekte aus einer vagen Idee. Der Factory-Worker koordiniert ai-agent (Spec + Code Gen), shell-agent (File I/O) und code-agent (Review).

Unterstützte Pipeline-Typen

  • app — Expo (React Native)
  • website — Next.js
  • mcp-server — MCP + Bun
  • agent — KI-Agent
  • api — REST API
  • cli — CLI-Tool
Ralph Mode (Quality Gate) Jede generierte Datei wird mehrdimensional bewertet. Bei zu niedrigem Score läuft automatisch eine Fix-Schleife — so lange, bis alle Qualitätskriterien erfüllt sind.

Memory & Knowledge

src/memory.ts schreibt Erinnerungen parallel in zwei Speicher:

  • SQLite — ~/.a2a-memory.db (schnelle Abfragen)
  • Obsidian Markdown — ~/Documents/Obsidian/a2a-knowledge/_memory/ (human-readable, versionierbar)

Jeder Worker kann remember und recall nutzen, um kontextuelles Wissen zu persistieren.

Distributed Tracing

src/tracing.ts implementiert OpenTelemetry-style Observability. Jede Delegation erstellt automatisch eine Trace mit Kind-Spans pro Worker-Aufruf.

  • Trace → Span Hierarchie mit automatischer Context-Propagation
  • Span Tags, Events, Status-Tracking
  • Waterfall-Visualisierung für Debugging
ToolBeschreibung
list_tracesAlle Traces auflisten
get_traceWaterfall-Ansicht einer Trace abrufen
search_tracesTraces nach Skill/Agent/Status suchen

Metriken

src/metrics.ts sammelt Skill-Ausführungsmetriken. Verfügbar via get_metrics-Tool und a2a://metrics-Resource.

  • Call-Counts pro Skill
  • Latenzen: p50, p95, p99
  • Fehlerquoten pro Worker

Skill Cache

src/skill-cache.ts ist ein LRU-Cache mit TTL für idempotente Skill-Ergebnisse.

  • Content-addressable Keys (deterministisches Hashing von skillId + args)
  • Per-Skill TTL-Konfiguration
  • Side-Effect-Skills (run_shell, write_file, etc.) werden automatisch ausgeschlossen
  • Integriert in den Delegate-Flow: Cache-Check vor Worker-Aufruf

Circuit Breaker

src/circuit-breaker.ts schützt das System vor kaskadierende Ausfällen.

  • Zustände: closed → open → half_open
  • Bei open: Fail-Fast ohne Worker-Aufruf
  • Auto-Recovery nach konfigurierbarem Cooldown
  • Ein Circuit Breaker pro Worker

Auth & RBAC

src/auth.ts verwaltet API-Keys mit SHA-256-Hashing und rollenbasierter Zugriffskontrolle.

RolleBerechtigungen

src/workspace.ts implementiert Team-Workspaces mit Mitgliedern, Rollen (owner/member/readonly), gemeinsamer Umgebung und Knowledge-Directories.

Audit Logging

src/audit.ts schreibt ein unveränderliches SQLite-Audit-Trail aller Skill-Invocations nach ~/.a2a-mcp/audit.db.

  • Abfragbar nach: Actor, Skill, Workspace, Zeitraum
  • Automatisch indiziert für schnelle Queries
ToolBeschreibung

Webhooks

src/webhooks.ts erlaubt die Integration externer Dienste (GitHub, Stripe, etc.).

  • HMAC-SHA256 Signaturverifizierung
  • Payload-Field-Mapping auf A2A-Tasks
  • Asynchrone Task-Erstellung
  • Endpunkt: POST /webhooks/:id auf dem A2A-HTTP-Server (Port 8080)

Federation

Remote-Worker können ohne lokalen Prozess eingebunden werden. Der Orchestrator entdeckt, health-pollt und routet Skills zu ihnen — genau wie bei lokalen Workern. 

// ~/.a2a-mcp/config.json
{
  "remoteWorkers": [
    { "url": "https://my-agent.example.com", "name": "custom-agent" }
  ]
}

Die URL wird automatisch in der SSRF-Validierung whitegelistet.

Plugin System

Das Plugin-System (src/skill-loader.ts) lädt benutzerdefinierte Worker aus ~/.a2a-mcp/workers/.

  • Auto-Discovery beim Start
  • File-Watcher für Hot-Reload im Entwicklungsmodus
  • Plugins können neue Skills, Personas und MCP-Tools bereitstellen
